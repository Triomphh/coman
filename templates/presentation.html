Architecture

Architecture du Projet

Une architecture client-serveur classique où le serveur C++ gère la logique métier et les données, tandis que le client web assure l'interface utilisateur.

  • • Serveur C++ avec Crow pour les API REST
  • • Client web en HTML/CSS
  • • Communication via requêtes HTTP

Architecture Serveur

Architecture en couches suivant le pattern MVC avec une séparation claire des responsabilités.

  • • View : Templates HTML pour le rendu des pages
  • • Controller : Gestion des requêtes et de la logique de présentation
  • • Service : Logique métier et orchestration des opérations
  • • Repository : Accès aux données et requêtes SQL
  • • Model : Classes représentant les entités métier

Avantages de l'Architecture

Cette architecture permet une séparation claire des responsabilités, rendant le code plus maintenable et évolutif. Les services encapsulent la logique métier et peuvent être réutilisés à travers différents controllers, tandis que les repositories isolent complètement l'accès aux données. Cette abstraction facilite les changements de base de données et permet d'écrire des tests unitaires efficaces grâce à l'injection de dépendances. La séparation entre les services et les repositories permet également de modifier la logique métier sans impacter la couche de données, et vice-versa.

Base de Données

Le projet utilise SQLite comme base de données, implémentée avec un pattern Singleton pour garantir une instance unique de la connexion. Cette approche assure une gestion efficace des ressources et évite les problèmes de concurrence d'accès à la base de données.

Diagramme de Classes

Le diagramme présente les principales classes du projet organisées selon notre architecture MVC. Chaque entité (Project, Task, User) possède son propre Controller qui gère les requêtes, son Service qui implémente la logique métier, et son Repository qui gère l'accès aux données. Cette structure en couches permet une séparation claire des responsabilités et facilite l'évolution du code.

Pipeline CI/CD

Processus de déploiement automatisé.

  • • Intégration continue
  • • Tests automatisés

pipeline {
    agent any

    environment {
        DOCKER_IMAGE = 'triomph/coman'
        VERSION = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
        CONTAINER_NAME = 'coman'
        VOLUME_NAME = 'coman-data'
        APP_PORT = '18080'
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
                sh 'git submodule update --init --recursive'
            }
        }

        stage('Build') {
            steps {
                script {
                    sh "docker build -t \${DOCKER_IMAGE}:\${VERSION} -t \${DOCKER_IMAGE}:latest ."
                }
            }
        }

        stage('Test') {
            steps {
                script {
                    sh "docker run --rm \${DOCKER_IMAGE}:\${VERSION} /bin/bash -c './coman_test'"
                }
            }
        }

        stage('Deploy') {
            steps {
                script {
                    sh "docker volume create \${VOLUME_NAME} || true"
                    sh "docker stop \${CONTAINER_NAME} || true"
                    sh "docker rm \${CONTAINER_NAME} || true"
                    sh """
                        docker run -d --name \${CONTAINER_NAME} \\
                            -v \${VOLUME_NAME}:/app/data \\
                            -p \${APP_PORT}:\${APP_PORT} \\
                            --restart unless-stopped \\
                            \${DOCKER_IMAGE}:\${VERSION}
                    """
                }
            }
        }
    }

    post {
        failure {
            script {
                sh "docker stop \${CONTAINER_NAME} || true"
                sh "docker rm \${CONTAINER_NAME} || true"
            }
        }
    }
}

Conteneurisation

Notre application est conteneurisée avec Docker pour assurer une portabilité et un déploiement cohérent. L'image est basée sur Ubuntu 24.04 et inclut toutes les dépendances nécessaires pour la compilation et l'exécution du projet C++. Le conteneur expose le port 18080 et maintient les données de l'application dans un volume persistant. 

FROM ubuntu:24.04

# Avoid interactive prompts during package installation
ENV DEBIAN_FRONTEND=noninteractive

# Install required packages
RUN apt-get update && apt-get install -y \
    build-essential \
    cmake \
    libsqlite3-dev \
    libasio-dev \
    libgtest-dev \
    && rm -rf /var/lib/apt/lists/*

# Set the working directory
WORKDIR /app

# Create data directory
RUN mkdir -p /app/data

# Copy the current directory contents into the container at /app
COPY . .

# Create build directory and build the project and tests
RUN rm -rf build && \
    mkdir build && \
    cd build && \
    cmake .. && \
    make && \
    make coman_test

# Expose the port that the application listens on
EXPOSE 18080

# Run the application
CMD ["./coman"]

TEST_F(ProjectRepositoryTest, CreateAndGetProject) 
{
    // Create a test project
    Project project = createTestProject();

    // Create the project in the database
    repo->create(project);

    // Get the project with ID 1 (first project)
    auto retrieved = repo->get(1);

    // Verify the project is correct
    ASSERT_TRUE(retrieved.has_value());
    EXPECT_EQ(retrieved->name, "Test Project");
    EXPECT_EQ(retrieved->description, "A test project");
    EXPECT_EQ(retrieved->start_date, "2024-01-01");
    EXPECT_EQ(retrieved->end_date, "2024-12-31");
}
